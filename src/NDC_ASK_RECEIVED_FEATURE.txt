## 1. Executive Summary

### 1.1 Problem Statement

Currently, when items are scanned during returns, the system fetches a `currentPrice` from the NDC Pricing Book. However, there is no intelligence around:
- What price to **ask** for from manufacturers
- What price to **expect to receive** based on historical data
- How to learn from credit memo payments to improve future pricing

### 1.2 Solution Overview

Implement an AI-powered pricing intelligence system that:
1. **Uploads credit memo PDFs** to the system
2. **Analyzes credit memos** using Azure AI Document Intelligence
3. **Extracts ask vs received values** per NDC from credit memos
4. **Stores historical payment data** in the database
5. **Calculates optimal ask prices** during return item scanning
6. **Updates pricing intelligence** in real-time as new data comes in

### 1.3 Key Benefits

- **Smarter Pricing:** Base ask prices on actual payment history
- **Better Predictions:** Know expected received amounts before processing
- **Self-Improving:** Each credit memo makes the system smarter
- **Risk Assessment:** Identify NDCs with poor payment history
- **Cash Flow Accuracy:** More realistic revenue forecasting

## 2. System Overview

### 2.1 High-Level Architecture

USER WORKFLOW:
SCAN ITEM -> GET PRICING INTELLIGENCE -> SHOW ASK & EXPECTED -> ADD TO RETURN

PRICING INTELLIGENCE ENGINE:
- NDC Pricing Book (Database) stores: NDC, Current Price, Ask Avg, Received Avg, Ratio %, Confidence %, Samples
- Learning Engine updates pricing from credit memo analysis

CREDIT MEMO PROCESSING:
UPLOAD PDF -> AZURE AI ANALYSIS -> EXTRACT VALUES -> UPDATE PRICING -> BROADCAST UPDATES

### 2.2 Data Flow

1. CREDIT MEMO UPLOAD FLOW:
   Admin Uploads PDF -> Azure AI Analyzes -> Extract NDC/Ask/Received -> Store in DB -> Broadcast Updates

2. SCANNING FLOW:
   User Scans Item -> Lookup NDC Intelligence -> Calculate Optimal Ask -> Show Ask/Expected -> Add to Return

## 3. Database Schema Changes

### 3.1 Enhance `ndc_pricing` Table

New Columns:
- avg_ask_price DECIMAL(12,2) - Average ask price from historical credit memos
- avg_received_price DECIMAL(12,2) - Average received price from historical credit memos
- ask_received_ratio DECIMAL(5,2) - Percentage: (avg_received / avg_ask) * 100
- payment_sample_count INTEGER - Number of credit memo payments analyzed
- ai_confidence DECIMAL(5,2) - AI analysis confidence percentage (0-100)
- manufacturer_reliability TEXT - Rating: excellent, good, average, poor, unknown
- last_ask_received_update TIMESTAMPTZ - When ask/received data was last updated
- min_received_price DECIMAL(12,2) - Lowest received price observed
- max_received_price DECIMAL(12,2) - Highest received price observed
- last_5_payments JSONB - JSON array of last 5 payment records

### 3.2 New Table: `ndc_payment_history`

Stores individual payment records for detailed analytics:
- id UUID PRIMARY KEY
- ndc VARCHAR(20) NOT NULL
- ndc_normalized VARCHAR(11) NOT NULL
- debit_memo_id UUID (references debit_memos)
- ask_price DECIMAL(12,2) NOT NULL
- received_price DECIMAL(12,2) NOT NULL
- payment_ratio DECIMAL(5,2) GENERATED (calculated)
- manufacturer TEXT
- product_name TEXT
- ai_extracted BOOLEAN
- ai_confidence DECIMAL(5,2)
- created_at TIMESTAMPTZ

### 3.3 New Table: `credit_memo_analysis`

Stores AI extraction results for audit and review:
- id UUID PRIMARY KEY
- debit_memo_id UUID (references debit_memos)
- credit_memo_url TEXT NOT NULL
- ai_status TEXT (pending, processing, completed, failed, manual_review)
- ai_confidence DECIMAL(5,2)
- ai_extracted_total DECIMAL(12,2)
- ai_extracted_items JSONB
- human_reviewed BOOLEAN
- human_approved BOOLEAN
- reviewed_by UUID
- created_at, updated_at TIMESTAMPTZ

### 3.4 RPC Functions

update_ndc_pricing_from_payment:
- Updates NDC pricing book with new ask/received data
- Calculates rolling averages
- Determines manufacturer reliability
- Maintains last 5 payments for trend analysis

get_ndc_pricing_intelligence:
- Retrieves complete pricing intelligence for an NDC
- Returns all pricing fields including historical data

## 7. Pricing Formula & Algorithm

### 7.1 Core Formula

OptimalAskPrice = TargetReceived / (HistoricalPayRate / 100) × AdjustmentMultiplier

Where:
- TargetReceived = What we want to receive (usually currentPrice)
- HistoricalPayRate = Average (received / ask) × 100 from historical data
- AdjustmentMultiplier = Risk-based adjustment factor

### 7.2 Adjustment Multiplier Table

Manufacturer Reliability:
- Excellent: +0%
- Good: +2%
- Average: +5%
- Poor: +12%
- Unknown: +8%

Sample Size:
- < 3 samples: +8%
- 3-5 samples: +4%
- 5-15 samples: +0%
- 15+ samples: -2%

AI Confidence:
- < 70%: +6%
- 70-80%: +3%
- > 80%: +0%

### 7.3 Risk Level Classification

Low Risk: Ratio >= 85% AND Samples >= 5 AND Confidence >= 80%
Medium Risk: Ratio >= 70% AND Samples >= 3
High Risk: Everything else

### 7.4 Example Calculations

Example 1: Reliable Manufacturer
- Target Received: $15.00
- Historical Pay Rate: 90%
- Manufacturer: Excellent, Samples: 20, AI Confidence: 92%
- Base Ask = $15.00 / 0.90 = $16.67
- Multiplier = 1.0 + 0% - 2% + 0% = 0.98
- Optimal Ask = $16.67 × 0.98 = $16.34
- Expected Received = $14.70
- Risk Level: LOW

Example 2: Poor Manufacturer
- Target Received: $15.00
- Historical Pay Rate: 65%
- Manufacturer: Poor, Samples: 2, AI Confidence: 68%
- Base Ask = $15.00 / 0.65 = $23.08
- Multiplier = 1.0 + 12% + 8% + 6% = 1.26
- Optimal Ask = $23.08 × 1.26 = $29.08
- Expected Received = $18.90
- Risk Level: HIGH

Example 3: New NDC (No History)
- Target Received: $15.00
- No historical data
- Default: 15% buffer
- Optimal Ask = $15.00 × 1.15 = $17.25
- Expected Received = $13.00 - $15.00 (range)
- Risk Level: HIGH

## 8. Real-Time Updates System

### 8.1 Server-Sent Events (SSE)

Backend: /src/controllers/sseController.ts
- Manages SSE connections
- Broadcasts pricing updates

Frontend: /Frontend/hooks/usePricingUpdates.ts
- Subscribes to pricing updates
- Triggers UI updates when pricing changes

### 8.2 Update Flow

1. Credit memo processed
2. Backend updates NDC pricing
3. Backend broadcasts SSE event
4. All connected clients receive update
5. If user is scanning same NDC, UI updates in real-time
