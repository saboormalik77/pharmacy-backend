================================================================================
  ROOT CAUSE
================================================================================

  The warehouse and batch operations (Main Admin) need to look up transaction
  data that lives in the BG Admin database. This is a genuine cross-DB
  dependency that exists in the original monolithic design.

  Key dependency: return_transactions + return_transaction_items (BG Admin)
                  are referenced by warehouse, batch, FedEx, and destruction
                  functions (Main Admin).

================================================================================
  OPTIONS TO FIX
================================================================================

  Option 1 — Move return_transactions to Main Admin DB
             Pros: All warehouse/batch functions work immediately
             Cons: BG Admin (pharmacy/processor) also needs these tables

  Option 2 — Duplicate the tables in both DBs and sync them
             Pros: Both DBs work independently
             Cons: Data sync complexity, risk of inconsistency

  Option 3 — Refactor these 11 RPC functions to accept transaction data
             as input parameters instead of querying return_transactions
             Pros: Clean separation, no cross-DB dependency
             Cons: Requires code changes in backend

  Option 4 — Keep these 11 functions only in the source DB and route
             warehouse/batch API calls to source DB for these endpoints
             Pros: No code changes needed
             Cons: Source DB still needed in production
